## 2. プロジェクトの目的

1. **完全分解** — 「私」を atomic な要素（事実・主張・仮説・概念・価値観・エピソード・人物・時期など）に分解し、一元管理する
2. **多面ビュー** — 同じ atomic 要素を複数の視点から束ね直し、いろいろな方面から「私」を見られるようにする
3. **持続可能な運用** — 新しい気づきが生まれたら、数分で atom として登録できる仕組みにする
4. **AI への引き継ぎ** — Claude/ChatGPT を継続的な対話相手として運用する前提で、構造化された人物像を提供する
5. **創作素材** — 小説（特にAMABレズビアン主人公のファンタジー）の世界観構築の素材にする
6. **発信素材** — YouTube・note 等での発信における整合性検証の素材にする

## 3. アーキテクチャ概要

「**完全分解**」と「**多面ビュー**」を、**データ層・ビュー層・タグ層の3層分離**で実現する。

```
ビュー層（views/）        … 視点別に束ねた読み物。参照のみで本文を持たない
   ↓ 参照
データ層（data/）         … 12タイプの atom の正本。一つの事実は一箇所だけに書く
   ↓ タグ付け
タグ層（tags/）           … 横断インデックス。自動生成
```

詳細は `_design/02_アーキテクチャ全体像.md` を参照。

## 8. 文体・トーンの方針

このフォルダ全体は本人が **「私（hoehoe）が私を語る」** スタイルで書かれている。Claude がこのフォルダを更新するときも以下を守ること：

- 一人称は **「私」**
- 文体は **「である」調と「です・ます」調の混在**を許容。**新規追加では「である」調を優先**
- **事実とデータを淡々と**並べることを優先する。感情的な表現は本人が書いたエピソード以外では抑制
- **「サンプル数1」「あくまで仮説」「現時点での結論」** という相対化のフレーズを大事にする
- **断定を避ける慎重さ** と **本人の発見への確信** が両立した語り口
- 学術用語（マズロー、System1/2、デフォルトモードネットワーク、Friston 予測符号化、Karl Lakoff、Piaget、ポライトネス理論など）を躊躇なく使ってよい。本人はそういう語彙圏にいる
- 機微情報（性同一性、うつ病歴、職歴の挫折、人間関係の失敗、家族構成など）は **隠さず正確に** 書いてよい。本人がそう望んでいる

### 8.1. 生得的特性の framing（重要・公開を意識した表現規約）

このプロジェクトは公開を前提とする。本人 hoehoe の三つの生得的特性（メイレズビアン／HSP／承認欲求の不在）について書くとき、**「強み」「特技」「武器」のように肯定的に持ち上げる表現を避ける**こと。

本人の認識は次の通り：

> **生得的にこういう体質を持って生まれたがゆえに、現状こうなってしまった**

これは強みでも欠陥でもなく、**構造的な出発点・与えられた条件**である。これを踏まえて、

- ❌ NG例：「三本柱」「強み」「武器」「個性的な才能」「ユニークな視点を持つ」
- ❌ NG例：「ほかの人にはない感受性を活かして」「特殊な能力を持っている」
- ✅ OK例：「三つの生得的特性」「先天的にこの構造を持っている」「この体質ゆえに〜となった」
- ✅ OK例：「47年間この特性とともに生きてきた結果として〜」「この生得的条件から派生して〜」

要するに、**因果的・構造的・記述的な framing** を使う。価値判断（良い／悪い、強い／弱い）を含めない。

この規約は、本人が書いた既存エピソードや当時の感覚の記述には適用しない（当時の語り口は保持する）。**新規に書く解説・サマリー・ビュー・atom 本文の地の文に対して適用する**。

詳細：`_design/07_運用ルール.md` の §4

## 9. 新しい気づきを追記するとき（移行後）

1. 気づきの**タイプを判定**する（EP / CL / HY / CO / VL / ... ）
2. 該当タイプのフォルダ `docs/data/<type>/` で次の ID を取得
3. 新規 atom ファイルを作成（`<プレフィックス>-XXX_<タイトル>.md`）
4. フロントマターを `_design/03_データ層スキーマ仕様.md` のスキーマに従って書く
5. 本文を書く
6. 関連 atom を最低1つ参照する
7. 関連 atom 側にも逆参照を追加
8. 必要なら関連ビューを更新（または「ビュー要更新」タグを付ける）
9. `LOG.md` に1行記録
10. `update.bat` で公開

判定に迷う場合：**まず EP として登録**。後で抽象化が進んだ段階で CL や RP に昇格させる。

詳細：`_design/07_運用ルール.md` の §1〜2

> **注記**：移行完了までは旧構造への追記も許可するが、できれば新構造で受ける。Phase 0〜2 の進捗は TASKS.md を参照。
