## 2026-05-08 — 体系化アーキテクチャの全面再設計（Phase 0 着手）

### 背景
2026-05-06 の初期セットアップ後、運用してみるとフォルダ構造に**整理軸の二重化**が発生していた。当初の8章構成（目的別）に加え、後から「特性別」（03_性同一性障害／04_HSP／05_承認欲求）の章が増設され、両者が並走。さらに `エピソード/` がフラット運用で独立。同じ事実があちこちに分散し、「どこに何を書けばよいか」を毎回判断するコストに消耗していた。

本人 hoehoe からの要望は次の二つに集約された：

1. **完全に分解できるぐらいしっかり体系化したい**
2. **いろいろな方面から見れるようにしたい**

これを満たすために、執筆と整理を分離する**3層アーキテクチャ（データ層・ビュー層・タグ層）**を採用することにした。

### 採用した設計

**データ層（atom 12タイプ）**：EP / CL / HY / CO / VL / FA / PP / TP / TH / IN / RP / BP の正本を一元管理。一つの事実は一箇所だけに書く。

**ビュー層（7軸）**：主題別／時系列／抽象度別／関係別／用途別／信頼度別／入口別。データ層を参照する読み物で、本文を持たない。

**タグ層**：データ層に貼られる横断インデックス。自動生成。

**オーケストレーション**：抽出・整合・ビュー・検証を専門エージェントに分担させる。実装手段は Claude のサブエージェント機能（Agent ツール / Subagents API）。

### 設計上の判断

- **整理軸の更新を、データ層 vs ビュー層の分離で構造的に解決**：今後新しい整理軸が生まれてもビュー層に並列追加するだけで済む
- **物理削除はしない**：旧構造は `_legacy/` または `_archive/` に退避させ、可逆性を担保
- **mkdocs + GitHub Pages の枠組みは丸ごと維持**：中身（`docs/` 配下）だけ3層構造に再編成
- **本人 hoehoe のレビューを各 Phase の完了条件にする**：AI に任せきりにせず、自己認識との一致を最終判断は本人が行う
- **オーケストレーションは Claude の Subagents 機能で実装**：今は単一 Claude セッション内で役割を切り替え、規模拡大時に Agent SDK へ移行
