# 07_運用ルール — 日々の追記とメンテナンス

> 本書は移行完了後（および移行中）の**日々の運用ルール**を定義する。
> ルールに従って動かしている限り、構造は壊れず、スケールしても破綻しない。

---

## 0. 運用の根本原則

1. **データ層への追記が最優先**。新しい気づきは必ず atom として登録する
2. **ビュー層は遅れて追従**。データ層が固まってからビューを直す
3. **機械検証を信じる**。`validate_data.py` が通らない atom は merge しない
4. **月次レビューを欠かさない**。整合性は時間で劣化するので定期点検が必要

---

## 1. 新しい気づきが生まれたとき

### 1.1. 判断フロー

```
[新しい気づき発生]
        │
        ▼
[これは何のタイプか？]
        │
   ┌────┼─────┬────┬────┬────┬────┬────┐
   ▼    ▼     ▼    ▼    ▼    ▼    ▼    ▼
   EP   CL    HY   CO   VL   FA   PP    その他
   │    │     │    │    │    │    │     │
   ▼    ▼     ▼    ▼    ▼    ▼    ▼     ▼
[該当タイプの atom を新規作成 または 既存を更新]
   │
   ▼
[関連 atom への双方向参照を追加]
   │
   ▼
[検証エージェント（validate_data.py）にかける]
   │
   ▼
[必要なら関連ビューを更新（または「ビュー要更新」タグを付ける）]
   │
   ▼
[LOG.md に短く記録]
   │
   ▼
[update.bat で公開]
```

### 1.2. タイプ判定の基準

| 気づきの形 | 該当タイプ |
| --- | --- |
| 「あのとき〜が起きた」 | EP |
| 「私は〜だ」と確信できる | CL |
| 「他者と私の違いは〜なのではないか」（説明仮説） | HY |
| 新しい用語、独自の意味で使う言葉 | CO |
| 「自分にとって〜が大事だ」 | VL |
| 客観的な事実（年齢、学歴、職歴など） | FA |
| 特定の人物の影響 | PP |
| 人生の時期区分 | TP |
| 既存理論の適用 | TH |
| 影響を受けた作品・人物 | IN |
| 反復する反応パターン | RP |
| 反復する行動パターン | BP |

判定に迷う場合の優先順位：**まず EP として登録**。後で抽象化が進んだ段階で CL や RP に昇格させる。

### 1.3. 追記の最小作業

新しい atom を1件追加する作業は次の通り。所要時間は5〜15分。

1. 該当フォルダで次の ID を取得（連番）
2. ファイル `<プレフィックス>-XXX_<タイトル>.md` を作成
3. フロントマターを設計書 03 のテンプレに従って書く
4. 本文を書く（atom タイプによる）
5. 関連 atom を最低1つ参照する
6. 関連 atom 側にも逆参照を追加する
7. 必要なら新規タグを `tags/` に追加
8. `validate_data.py` を実行
9. `LOG.md` に1行記録
10. commit & push

## 2. ID 採番ルール

### 2.1. 連番の採番

- 既存の最大 ID を確認し、+1 する
- 削除済み ID は再利用しない
- 偶発的な重複を避けるため、`hooks/next_id.py` を用意（将来）

### 2.2. ID の桁

- 初期は3桁（EP-001 〜 EP-999）
- 1000 を超えたら4桁（EP-1000）に切り替え
- 既存 ID は 3桁→4桁に変換する必要はない（EP-001 のまま）

### 2.3. ファイル名と ID

- ファイル名は `<ID>_<簡潔な日本語タイトル>.md`
- タイトル部分は OS の検索性のため。中身の id がフロントマターと一致していることが正
- タイトルに特殊記号（`/` `\` `:` `*` `?` `"` `<` `>` `|`）は使わない

## 3. タグ統制

### 3.1. タグは統制語彙

タグは自由記述ではなく、**事前に定義した語彙**から選ぶ。新タグを増やすときは:

1. `_design/` または `tags/` の該当ファイルに新タグを追加
2. 採用理由を1行記録
3. 既存 atom で該当しそうなものに反映

### 3.2. タググループ

| グループ | 値（初期セット） |
| --- | --- |
| 特性 | メイレズビアン / HSP / 承認欲求 / 合理性駆動 / ヒューマニズム / 4段階目薄 |
| 時期 | 〜15歳 / 15-24歳 / 24-28歳 / 28-37歳 / 37-46歳 / 46歳〜 |
| レンズ | 哲学 / 社会 / 心理 / 言語 / 性 / 創作 / AI / 健康 / 経済 |
| 信頼度 | 確信 / 仮説 / 推測 / 保留 |
| 情動 | 強い後悔 / 嫌悪 / 解放 / 違和感 / 恐怖 / 諦め / 興奮 / 安堵 |
| 領域 | 履歴書 / 創作 / 対人 / 自己 / 身体 |

### 3.3. タグ整理ルール

- 1 atom に貼るタグは **3〜7個** 程度を目安
- 多すぎると検索が機能しない、少なすぎるとビューに引っかからない
- タグは年に1回見直す（後述：年次メンテナンス）

## 4. 文体と語り口の規約

`SelfAnalysis/CLAUDE.md` のトーン規約を再掲・強化する。

### 4.1. 一人称
- **「私」** で統一。「ぼく」「自分」は使わない（必要な場面では引用として可）

### 4.2. 文体
- 新規追記は **「である」調を優先**
- 既存の「です・ます」調を一括変換しない（推敲時期の証跡として残す）
- 説明部分（atom の本文）は「である」調、エピソードの「当時の感覚」部分は当時の語り口を保持してよい

### 4.3. 留保フレーズ
次のフレーズを大事にする：
- 「サンプル数1」
- 「あくまで仮説」
- 「現時点での結論」
- 「他の可能性も否定しない」

### 4.4. 学術用語の取り扱い
- 学術用語（マズロー、System1/2、デフォルトモードネットワーク、Friston 予測符号化、Lakoff フレーム理論、Piaget、ポライトネス理論）は躊躇なく使ってよい
- ただし、初出のときは TH（外部理論 atom）への参照を必ず張る
- 用語の定義を本文に書かない（TH に任せる）

### 4.5. 機微情報
- 隠さず正確に書く（性同一性、うつ病歴、職歴の挫折、人間関係の失敗、家族構成）
- ただし、第三者の名前・特定可能な情報は抑える（PP は役割で識別）
- 機微度の高い atom には `category: 機微` タグを付ける（公開範囲の制御に使う）

### 4.6. 感情表現
- 分析的な記述を基調とする
- 当時感じた感情は「当時の感覚」として埋め込む形に限定
- 現在の解釈節では分析的に距離を取る

### 4.7. 生得的特性の framing（重要・公開を意識した表現規約）

このプロジェクトは公開を前提とする。本人 hoehoe の三つの生得的特性（メイレズビアン／HSP／承認欲求の不在）について書くとき、**「強み」「特技」「武器」のように肯定的に持ち上げる表現を避ける**こと。

本人の認識は次の通り：

> **生得的にこういう体質を持って生まれたがゆえに、現状こうなってしまった**

これは強みでも欠陥でもなく、**構造的な出発点・与えられた条件**である。

#### 表現の指針

| ❌ 避ける | ✅ 推奨 |
| --- | --- |
| 三本柱 | 三つの生得的特性 |
| 強み・武器・特技 | 生得的特性・先天的条件・体質 |
| 〜を活かして | 〜という構造ゆえに |
| ユニークな才能 | 生得的にこの構造を持つ |
| 〜が得意 | 〜と感じる傾向がある |
| 個性的・特殊な能力 | 多数派と異なる構造 |

#### 因果の方向

- ❌ 「この特性のおかげで〜できる」（特技・能力としての持ち上げ）
- ✅ 「この特性ゆえに〜となっている」（構造から現状への因果）
- ✅ 「47年間この体質とともに生きてきた結果〜」（時間的・累積的な記述）

#### 適用範囲

- **適用する**：新規に書く解説・サマリー・ビュー・atom 本文の地の文
- **適用しない**：本人が書いた既存エピソード、当時の感覚の引用、本人が当時用いた言葉
  （当時の語り口は史料として保持する）

#### この規約が必要な理由

本人は公開を前提としており、自身の特性を「特技」のように見せたくないという意向を持つ。
価値判断（良い／悪い、強い／弱い）を含めず、**因果的・構造的・記述的な framing** で書く。これにより、

- 同じ特性を持つ他者が「自分は強みではない」と感じても疎外されない
- 多数派の読者に「特別な人の話」として消費されずに、構造的な事実として伝わる
- 本人が後で読み返したときに違和感が出にくい

## 5. 月次運用

毎月1回、以下のメンテナンスを行う。所要時間：1〜2時間。

### 5.1. データ層の点検
- [ ] 追加された atom の数を集計（各タイプ）
- [ ] `validate_data.py` を実行、エラーがあれば修正
- [ ] 双方向参照の整合性チェック
- [ ] 「ドラフト」「保留」状態の atom をレビュー、進められるものは進める

### 5.2. ビュー層の更新
- [ ] 新規 atom がある場合、関連ビューに反映
- [ ] 「ビュー要更新」タグが付いた atom を処理
- [ ] 自動生成系のビュー（全エピソード索引、用語集など）を再生成

### 5.3. タグ層の整理
- [ ] 新規タグの増減を確認
- [ ] 使用頻度の極端に低いタグを統合
- [ ] タグページ（自動生成）の見栄え確認

### 5.4. TASKS.md / LOG.md の更新
- [ ] 完了したタスクを LOG.md に移す
- [ ] 新しいタスク・問いを追加
- [ ] 次月の優先度を整理

### 5.5. 設計書の点検
- [ ] 運用してみて違和感のある設計箇所を `_design/` で改訂
- [ ] 改訂時は前バージョンを `_design/_archive/YYYY-MM-DD_XX.md` に退避

### 5.6. 月次レビューのテンプレート

`_design/templates/月次レビュー_YYYY-MM.md` を運用上のテンプレートとして用意する。

```markdown
# YYYY-MM 月次レビュー

## データ層の追加
- EP: +X件
- CL: +X件
- HY: +X件
- ...

## 機械検証の結果
- 通過: ○
- エラー: 件数 / 内容

## 整合上の課題
- 重複候補: ...
- 矛盾検出: ...

## ビュー更新
- 更新: ...
- 新設: ...

## 次月の優先
- ...

## 設計書の改訂
- ...
```

## 6. 年次運用

毎年1回、構造的なメンテナンスを行う。所要時間：1〜2日。

### 6.1. タグ語彙の見直し
- 1年間で増えたタグを整理
- 統合・廃止の判断
- タググループ自体の再編

### 6.2. ビュー軸の見直し
- 1年間で生まれた新しい興味から、新ビュー軸を追加
- 使われていないビューの統廃合

### 6.3. アーキテクチャ全体のレビュー
- 設計書 02（アーキテクチャ全体像）の見直し
- スキーマの拡張・変更（破壊的変更は最小限に）

### 6.4. 公開方針の見直し
- 機微情報の取り扱い方針の更新
- 公開範囲の調整

## 7. スケール戦略

データ量が増えたときに破綻しないために、以下を準備する。

### 7.1. atom 数が 100 を超えたタイプ

各タイプ（EP / CL など）が 100 件を超えたら：
- そのタイプ専用の **目次ファイル**（`docs/data/<type>/index.md`）を自動生成する
- 50件ずつバッチ化して目視確認できるようにする
- 命名規則を見直し、サブカテゴリ・サブプレフィックス（例：EP-性同一性-001）を検討

### 7.2. atom 総数が 1000 を超えたとき

- ID 桁の拡張（3桁→4桁）
- 検索用の全文インデックスを用意
- ビュー層の `index.md` を自動生成化

### 7.3. ビュー数が 30 を超えたとき

- ビュー軸ごとの index ページを自動生成
- ビュー間の重複検出を整合エージェントの定期業務に追加

### 7.4. 永続化と外部化

将来的に：
- atom データを **JSON エクスポート可能**にする（mkdocs ビルド時）
- 外部ツール（Obsidian、Logseq 等）との連携を検討
- 機微情報を分離した「公開版」「全部入り版」の二系統運用

## 8. AI 利用の運用ルール

### 8.1. AI に対するメタ情報の維持

- `CLAUDE.md` を常に最新に保つ
- 重要キーワード辞書を `views/用途別/AI引き継ぎビュー.md` に置く
- 月次で AI 引き継ぎビューを再生成

### 8.2. AI 利用の作業セッションでの注意

- 作業前に **設計書を読ませる**（_design/ 全体）
- 作業中に **CLAUDE.md のトーン規約を遵守させる**
- 作業後に **本人レビュー**を必ず挟む

### 8.3. AI が誤った参照を作ったとき

- 整合エージェントが検出し、引き継ぎメモに記録
- 月次レビューでまとめて修正
- 同じ誤りが繰り返される場合、CLAUDE.md に注意書きを追加

## 9. 個別の運用シナリオ

### 9.1. シナリオ：新しい考察ノートが MyConsiderations に増えた

1. ノートを読み、本質的な気づきがあるか判断
2. ある場合、CL / HY / VL のいずれかとして atom 化
3. 既存 EP / CL / HY との関連を張る
4. 該当ビューを更新

### 9.2. シナリオ：過去のエッセイに新しい解釈が浮かんだ

1. 該当 EP の本文の「後から見るとどう解釈できるか」節を更新
2. `updated` 日付を更新
3. 関連する CL / HY が変わるなら、それも更新

### 9.3. シナリオ：新しい人物が人生に登場した

1. PP-XXX を新規作成
2. その人物との出来事を EP として登録
3. 関係するビュー（私と他者など）を更新

### 9.4. シナリオ：仮説が反証された

1. 該当 HY の `status` を `保留` または `廃版` に変更
2. `conflicting_evidence` に反証を追記
3. 関連 CL の confidence を見直し
4. 月次レビューで影響範囲を確認

### 9.5. シナリオ：機微情報を一時的に非公開にしたい

1. 該当 atom のフロントマターに `published: false` を追加
2. ビューから該当 atom への参照は残す（タイトルのみ表示、本文非公開）
3. mkdocs のビルド時に `published: false` を除外する hook を実装

## 10. 緊急対応

### 10.1. 機微情報の誤公開

1. 該当ファイルを即座に削除（git rm）
2. `git push --force-with-lease` で公開ブランチを上書き
3. GitHub の cache を invalidate
4. `_archive/incidents/YYYY-MM-DD_誤公開.md` に経緯を記録

### 10.2. 構造の致命的破壊

1. 直近の正常 commit に `git reset --hard`
2. 設計書 06（移行計画）の「ロールバック方針」に従う
3. 失敗の原因を `LOG.md` に記録、運用ルールを改訂

## 11. 運用上の禁止事項

| 禁止事項 | 理由 |
| --- | --- |
| atom の物理削除 | 履歴を辿れなくなる。`status: 廃版` で残す |
| ID の再利用 | 過去の参照が壊れる |
| 設計書の改訂を経ずに `docs/` 構造を変える | 設計と実装のズレを生む |
| 検証エージェントを通さずに commit | 形式エラーが蓄積する |
| 機微情報を `_design/` に書く | 設計書は公開対象になりうるため |
| ビュー層に新規本文を書く | DRY 原則の違反 |
| 整合エージェントが新規 atom を作る | 整合は整理だけが役割 |

## 13. 誤情報の発見と訂正

過去に AI（Claude / ChatGPT）が捏造した可能性のある内容、本人レビュー時に発見された誤り、後から判明した事実誤認を、**確実に検出して直す仕組み**。本人 hoehoe からの「あとで治せるか」という懸念に応えるための運用節。

### 11.1. 三段階の防御線

| 段階 | 何をするか | 担当 |
| --- | --- | --- |
| 1. 自動検出 | `validate_data.py` が AI 捏造パターンをフラグ立てし、`verification_status: AI推測_要検証` に自動降格 | 検証エージェント |
| 2. レビュー検出 | 月次レビューで `verification_status: 未検証 / AI推測_要検証` の atom を本人がチェック | 人間 hoehoe |
| 3. 利用時検出 | atom を引用するときに違和感を覚えたら、その場で `verification_status: AI推測_要検証` に降格 + 修正キューに追加 | 利用者（本人 / Claude） |

### 11.2. 誤情報を発見したときのフロー

```
[誤情報・捏造・不足を発見]
        │
        ▼
[該当 atom の verification_status を変更]
   ・誤りが疑わしい  → AI推測_要検証
   ・明らかに誤り    → 削除推奨
   ・誤りだが訂正可能 → 訂正済（訂正後）
        │
        ▼
[TASKS.md の修正キューに1行追加]
   - [ ] EP-XXX：内容の要約 → 訂正必要
        │
        ▼
[該当 atom を引用しているビューに「修正中」フラグを付ける（任意）]
        │
        ▼
[月次レビューでまとめて訂正処理]
   ・訂正内容を本文に反映
   ・correction_history に履歴を残す
   ・verification_status: 訂正済 に変更
   ・修正キューの該当行を [x] でマーク
        │
        ▼
[必要なら関連 atom / ビューを連鎖更新]
```

### 11.3. correction_history の書き方

訂正したら、必ず atom フロントマターの `correction_history` に履歴を残す。

```yaml
correction_history:
  - date: 2026-06-15
    changed: age_range
    from: "17歳"
    to: "19歳"
    reason: 本人ヒアリングで実際は19歳と判明
  - date: 2026-07-02
    changed: 本文段落2
    from: "...3年続いた..."
    to: "...4年続いた..."
    reason: 簡易年表との照合で4年が正しいと判明
```

これにより、

- どの記述が訂正前のままビューに参照されていたかを追跡可能
- 過去に同じ箇所で誤りが繰り返されていないかを検出可能
- 本人が「以前読んだときと違う気がする」と思ったとき、変遷を辿れる

### 11.4. 削除推奨 → 廃版 へのフロー

`verification_status: 削除推奨` がついた atom は、

1. 一度ビュー層から参照を外す（リンクをコメントアウト）
2. 月次レビューで本人が判断
3. 本当に取り下げるなら `status: 廃版` に変更
4. 物理ファイルは残す（`_archive/廃版_YYYY-MM/` に移動）

物理削除しないので、**間違って取り下げてもいつでも復活できる**。

### 11.5. AI 捏造の典型パターン（チェックリスト）

検証エージェントが検出する代表パターン。本人手動レビュー時にも使える。

- [ ] 過度に整った文章（本人の語り口と乖離）
- [ ] 出典のない断定的記述
- [ ] 不自然に細かい数値（「31.5歳」「214冊」など）
- [ ] `source` フィールドが空または「不明」
- [ ] PP に本人未確認の固有名詞
- [ ] `confidence: 仮説` なのに本文で「明らかに」「間違いなく」を使用
- [ ] 引用ブロック（`>`）の出典ファイルが実在しない
- [ ] 三本柱・強み・武器など NG 表現の出現
- [ ] 本人が当時使っていなかった現代的な学術用語の混入

詳細：`_design/05_オーケストレーション運用設計.md` の §2.6

### 11.6. 「訂正したくないが残しておく」ケース

過去の認識自体が時系列の証拠として価値を持つ場合がある（例：当時はそう信じていた）。
そういうケースは、訂正せずに `confidence: 推測` または `status: 保留` にして、本文に「現在の認識との差分」を注記する。

例：

```markdown
## 注記
本 atom は当時の認識を記録したものである。後の検証で〜の点が事実と異なることが判明した。
当時の認識自体に資料的価値があるため、訂正せず保留として残す。
詳細は CL-XXX を参照。
```

過去の認識が変遷したこと自体が、自己理解の発展史の重要な記録になる。
