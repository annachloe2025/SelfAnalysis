# 02_アーキテクチャ全体像 — 3層構造と多面ビュー

> 本書は SelfAnalysis プロジェクトの**全体構造**を定義する。
> データ層・ビュー層・タグ層の3層と、それらの相互関係を示す。
> 設計書群の中核であり、後続の設計書（03〜07）はすべて本書のサブ仕様である。

---

## 0. 設計の根本原理

> **「私という人間」を atomic な要素に完全分解し、それを多面から組み立て直す。**

この一文がアーキテクチャ全体の指針である。要点は二つ。

1. **完全分解** — 一つの事実は一つの場所にしか書かない（DRY: Don't Repeat Yourself）
2. **多面組立** — 同じ事実を違う角度から見るための「ビュー」を独立に増やせる

これを実現するために、**データ層・ビュー層・タグ層**の3層を分離する。

---

## 0.5. 設計の学術的系譜

このアーキテクチャは独立に発明したものではなく、複数の確立された手法を**個人記述（自己分析）の文脈に応用**したものである。本書を読む第三者がどの先行知識の延長線上にあるかを把握できるよう、系譜を明記しておく。

### 0.5.1. データ層（atomic 単位の正本管理）

**主たる先行：ツェッテルカステン（Zettelkasten）**

ドイツの社会学者ニクラス・ルーマン（Niklas Luhmann, 1927-1998）が一生に9万枚以上のメモカードで運用した知識管理手法。
- 一つのカードに一つの atom（概念・事実・引用）を記述
- カード同士をユニーク ID で相互参照
- カテゴリ階層ではなく、リンクと文脈で意味を構成

本プロジェクトの 12 atom タイプ（EP/CL/HY/CO/VL/FA/PP/TP/TH/IN/RP/BP）と双方向参照の仕組みは、ツェッテルカステンの拡張である。タイプを 12 に増やしたのは、自己分析という用途特化のため。

**補助的先行：データベース正規化理論**

E.F. Codd（1970）に始まる関係データベースの設計原理。一つの事実は一箇所にしか書かない（第3正規形）という DRY 原則は、本プロジェクトの「正本一元管理」の理論的根拠となる。

### 0.5.2. atom タイプの分類根拠

各 atom タイプは別々の学術領域から借用している。

| atom | 借用元の学術領域 |
| --- | --- |
| Episode（EP） | 自伝的記憶研究（Tulving 1972, Conway & Pleydell-Pearce 2000） |
| Claim（CL） | 命題論理／認識論 |
| Hypothesis（HY） | 科学哲学（Hempel, Popper の仮説検証構造） |
| Concept（CO） | オントロジー工学（Gruber 1993）／概念分析 |
| Value（VL） | 価値倫理学（Schwartz 1992 の価値観研究） |
| Fact（FA） | 客観的伝記情報（履歴書・年表の標準項目） |
| Person（PP） | ソーシャル・ネットワーク分析の単位 |
| Period（TP） | ライフスパン心理学（Erikson の発達段階を参考） |
| Theory（TH） | 引用情報学（外部理論への参照管理） |
| Influence（IN） | 発達心理学（影響源の追跡） |
| Reaction（RP）／Behavior（BP） | パーソナリティ心理学の行動・反応傾向 |

12タイプの**取り合わせ自体は本プロジェクトのオリジナル**だが、各タイプの設定は確立された学問領域に根拠がある。

### 0.5.3. データ層から atom を抽出する手法

**主たる先行：グラウンデッド・セオリー・アプローチ（Glaser & Strauss 1967）**

質的研究で用いられる、データから帰納的にカテゴリを立ち上げる手法。
- オープンコーディング（事実をラベル化） ≒ EP抽出
- 軸足コーディング（カテゴリ間の関係付け） ≒ CL/HY抽出と相互参照
- 選択的コーディング（中核カテゴリへの統合） ≒ ビュー層構築

本プロジェクトの「素材から atom を起こす → 関係を張る → ビューに統合する」の3段階は、この手法の応用である。

### 0.5.4. ビュー層（多面的な物語構築）

**主たる先行：ナラティブ・アイデンティティ理論（McAdams 1985, 2001）**

「自己とは複数の物語の束である」とする心理学的立場。同一の事実が、誰にどの場面で語るかで違う物語になる。
- 主題別ビュー、入口別ビュー、用途別ビューはこの原理に依拠する
- 同じ EP が複数のビューに登場することの理論的正当性

**補助的先行：イディオグラフィック研究（Allport 1937）**

個人の独自性を一般法則ではなく**その個人特有の構造**で記述する研究流派。
本プロジェクトは典型的なイディオグラフィック研究のスタイルを取る。「サンプル数1」の留保は、この立場と整合する。

### 0.5.5. タグ層（横断インデックス）

**主たる先行：ファセット分類法（Ranganathan 1933, Colon Classification）**

複数の独立した次元（ファセット）でドキュメントを索引する分類学の手法。
- 特性タグ／時期タグ／レンズタグ／信頼度タグ／情動タグの並列構造はファセット分類

### 0.5.6. オーケストレーション

**主たる先行：マルチエージェント・システム（MAS）の役割分担パターン**

加えて、Claude の Subagents 機能（Anthropic 2025-2026）が技術的な実装基盤となる。

### 0.5.7. このプロジェクトのオリジナルな部分

確立された手法から借りていない、本プロジェクトの新規性は次の3点：

1. **自己分析という用途に特化した 12 atom タイプの取り合わせ**
2. **3層分離を mkdocs + GitHub Pages の標準的な静的サイト生成と統合**
3. **オーケストレーション設計を、Claude のサブエージェント機能を使って素人でも運用できる粒度で記述**

この三点を除けば、本アーキテクチャは「**確立された手法を、自己分析という個人の用途に組み合わせ直したもの**」と理解してよい。

## 2. データ層 — 「正本」の置き場所

### 2.1. 原則

- **一つの事実は一つの atom に正本がある**。本文はその atom にのみ書かれる
- atom には**永続 ID**（EP-001、CL-007 など）を振る。ID は二度と再利用しない
- atom 同士は**相互参照**でつながる（EP-002 が CL-014 を支える、など）
- atom は YAML フロントマター付き Markdown として保存し、機械的にも人間にも読める

### 2.2. atom の種類（12タイプ）

| 種類 | プレフィックス | 役割 | 例 |
| --- | --- | --- | --- |
| Episode | EP | 具体的に起きた一回の出来事 | EP-002「走り方が女の子みたいと言われた」 |
| Claim | CL | 著者が抱く命題 | CL-001「私には承認欲求がほとんどない」 |
| Hypothesis | HY | 自家製の仮説 | HY-001「9年×2サイクル仮説」 |
| Concept | CO | 造語・独自定義の用語 | CO-001「メイレズビアン」 |
| Value | VL | 価値観の単位 | VL-001「合理性」 |
| Fact | FA | 履歴書相当の客観事実 | FA-001「1977年生まれ」 |
| Person | PP | 関係者・登場人物 | PP-001「母」 |
| Period | TP | 人生の時期区分 | TP-002「新聞奨学生期（15-24歳）」 |
| Theory | TH | 外部の理論枠組み | TH-001「マズローの欲求段階説」 |
| Influence | IN | 影響を受けた作品・人物・体験 | IN-001「エヴァンゲリオン」 |
| Reaction | RP | 反応パターン | RP-001「SNSのいいねを気持ち悪く感じる」 |
| Behavior | BP | 行動パターン | BP-001「格闘ゲームで一貫して女性キャラを選ぶ」 |

詳細スキーマは `03_データ層スキーマ仕様.md` で定義する。

### 2.3. ファイル配置

```
data/
├── episodes/        EP-001.md, EP-002.md, ...
├── claims/          CL-001.md, ...
├── hypotheses/      HY-001.md, ...
├── concepts/        CO-001.md, ...
├── values/          VL-001.md, ...
├── facts/           FA-001.md, ...
├── people/          PP-001.md, ...
├── periods/         TP-001.md, ...
├── theories/        TH-001.md, ...
├── influences/      IN-001.md, ...
├── reactions/       RP-001.md, ...
└── behaviors/       BP-001.md, ...
```

### 2.4. 重要な性質

- データ層に**章という概念は存在しない**。章はビュー層の概念
- データ層には**「読みやすさ」を求めない**。一貫性と網羅性が最優先
- 新しい気づきが生まれたら、まず該当タイプの atom を新規作成または更新する
- ビュー層から見てつまらない atom（短い、地味）も削除しない。多面ビューの素材になる

## 3. ビュー層 — 「読み物」の編成

### 3.1. 原則

- ビューは**データ層の atom を参照（リンク・引用）するだけ**で本文を持たない
- 同じ atom が**複数のビューに登場してよい**。重複を恐れない
- ビューの追加・削除はデータ層に影響しない
- ビューには**目的・想定読者・読了時間**を明示する

### 3.2. ビュータイプ（7軸）

| 軸 | 説明 | ビューの例 |
| --- | --- | --- |
| 主題別 | 三つの生得的特性その他のトピック軸 | 「メイレズビアンレンズ」「HSPレンズ」「承認欲求レンズ」 |
| 時系列 | 時間軸 | 「9年×2サイクル」「簡易年表」「ライフフェーズ伝記」 |
| 抽象度別 | データタイプ別の俯瞰 | 「全エピソード索引」「全主張カタログ」「全仮説マップ」 |
| 関係別 | 私と何かの関係軸 | 「私と社会」「私と他者」「私と物語」「私と AI」 |
| 用途別 | 読まれ方の用途 | 「履歴書ビュー」「初対面用自己紹介」「同類への手紙」「AI引き継ぎ」 |
| 信頼度別 | 確信レベル | 「確信していること」「仮説の段階」「まだわからないこと」 |
| 入口別 | 初見の読者の興味から | 「性同一性から入る」「思想から入る」「キャリアから入る」「趣味から入る」 |

詳細は `04_ビュー層カタログ.md` で定義する。

│   └── ...

## 4. タグ層 — 横断インデックス

### 4.1. 原則

- タグは atom に**フロントマターで付与**する
- タグの種類は**統制された語彙**（あらかじめ決めたリストから選ぶ）
- 自由記述タグは原則禁止。新タグを増やすときは `tags/` 配下に定義ファイルを増やす
- タグは**ビューを自動生成する材料**にもなる

### 4.2. タグの体系

| タググループ | 値の例 |
| --- | --- |
| 特性タグ | `メイレズビアン` `HSP` `承認欲求` `合理性駆動` `ヒューマニズム` |
| 時期タグ | `〜15歳` `15-24歳` `24-28歳` `28-37歳` `37-46歳` `46歳〜` |
| レンズタグ | `哲学` `社会` `心理` `言語` `性` `創作` `AI` `健康` |
| 信頼度タグ | `確信` `仮説` `推測` `保留` |
| 情動タグ | `強い後悔` `嫌悪` `解放` `違和感` `恐怖` `諦め` |

### 4.3. ファイル配置

```
tags/
├── 特性.md      … 各特性タグの定義と該当 atom 一覧（自動生成）
├── 時期.md
├── レンズ.md
├── 信頼度.md
└── 情動.md
```

タグ定義ファイルは**自動生成可能**。データ層全体をスキャンし、各タグの該当 atom をリスト化する。

## 6. 3層の相互関係（具体例で確認）

「走り方が女の子みたいと言われた」というエピソードを例に、3層の連携を示す。

### データ層

`data/episodes/EP-002.md`（正本：本文はここだけに書く）

```yaml
---
id: EP-002
title: 走り方が「女の子みたい」と何度も言われた
age_range: 7-15
period: TP-001  # 義務教育期
related_traits:
  - メイレズビアン
related_claims:
  - CL-003  # 性自認が女性であることの手がかり
related_concepts:
  - CO-001  # メイレズビアン
confidence: 確信
情動: 違和感
source: 自身のエッセイ/01_性同一性障害メイレズビアン/01-1_メイレズビアンとは/01-1-2_走り方について指摘されていたこと.md
tags:
  - メイレズビアン
  - 〜15歳
  - 性
---

# EP-002 走り方が「女の子みたい」と何度も言われた

（本文はここに一回だけ書く）
```

### タグ層（自動生成）

`tags/特性.md` の「メイレズビアン」セクションに自動的に EP-002 が現れる。
`tags/時期.md` の「〜15歳」セクションにも現れる。

### ビュー層（複数のビューが EP-002 を参照する）

- `views/主題別/メイレズビアンレンズ.md` → EP-002 を引用
- `views/時系列/9年2サイクル.md` → 「学童期の身体違和感」の章で EP-002 を引用
- `views/関係別/私と身体.md` → EP-002 を引用
- `views/抽象度別/エピソード索引.md` → EP-002 をリストに含む

**EP-002 自体の本文は1つしかない。各ビューはリンクで参照する**。

## 7. このアーキテクチャがもたらす効果

| 課題 | 解決 |
| --- | --- |
| 同じ事実があちこちに分散して齟齬を生む | atom 正本制で重複なし |
| どこに何を書けばよいか毎回判断するコスト | atom タイプを選ぶだけ。ビューは後で組む |
| 整理軸が増えると構造が崩れる | ビュー層で並列に追加できる |
| AI が誤った参照を作る | atom ID で正確に引き継げる |
| 大規模化したときに読み切れない | ビュー単位で読める。素材は索引から辿れる |
| 47歳以降の新しい気づきの追加 | 該当タイプの atom を1つ増やすだけ |
