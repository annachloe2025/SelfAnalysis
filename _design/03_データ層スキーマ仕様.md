# 03_データ層スキーマ仕様 — 12タイプの atom を定義する

> 本書はデータ層を構成する 12 種類の atom（Episode / Claim / Hypothesis / Concept / Value / Fact / Person / Period / Theory / Influence / Reaction / Behavior）について、フィールド・命名規則・運用上の注意・具体例を定義する。
> 抽出エージェントと整合エージェントは本書を機械的な仕様として参照する。

---

## 0. 全タイプ共通の規約

### 0.1. ファイル形式

- すべて Markdown（`.md`）
- 先頭に YAML フロントマターを置き、本文を続ける
- ファイル名は `<プレフィックス>-<3桁ゼロ詰め番号>_<簡潔な日本語タイトル>.md`
  - 例：`EP-002_走り方が女の子みたいと言われた.md`
  - 簡潔な日本語タイトルは検索性のために付ける（OS 上で目視できる）

### 0.2. ID 規約

- ID は **永続不変**。一度振ったら絶対に変えない。削除した場合も再利用しない
- 番号は登録順に付与。意味のある番号（00X が重要等）はつけない
- 番号は 999 を超えたら桁を増やす（EP-1000）。容量の上限は実質ない

### 0.3. フロントマター共通フィールド

```yaml
id: <プレフィックス-番号>     # 必須・永続不変
title: <タイトル>             # 必須・1行
status: ドラフト | 整理済 | 保留 | 廃版                      # 必須・編集段階
confidence: 確信 | 仮説 | 推測 | 保留                        # 必須・内容の確信度
verification_status: 未検証 | 本人確認済 | AI推測_要検証 | 訂正済 | 削除推奨  # 必須・事実検証の段階
created: YYYY-MM-DD          # 必須・作成日
updated: YYYY-MM-DD          # 必須・直近更新日
source:                      # 必須・元素材へのパス（空にしない）
  - 自身のエッセイ/.../XX.md
correction_history:          # 任意・訂正履歴
  - date: YYYY-MM-DD
    changed: <フィールド名>
    from: <旧値の要約>
    to: <新値の要約>
    reason: <理由>
tags:                        # 必須（空配列でも可）
  - <タグ>
```

### 0.3.1. status / confidence / verification_status の役割の違い

3つの似たフィールドは役割を分けている。

| フィールド | 何を表すか | 例 |
| --- | --- | --- |
| `status` | **編集段階**（書き終わっているか） | ドラフト → 整理済 |
| `confidence` | **内容の確信度**（本人がどれだけ確信しているか） | 確信 → 仮説 → 推測 |
| `verification_status` | **事実検証の段階**（誤情報・捏造の有無を確認したか） | 未検証 → 本人確認済 |

例：「九年×二サイクル仮説」HY-002 は

- `status: 整理済`（文章は完成）
- `confidence: 仮説`（自分用説明仮説）
- `verification_status: 本人確認済`（年数・年齢に誤りがないか本人が確認した）

の組み合わせになる。

### 0.3.2. verification_status の各値の意味

| 値 | 意味 | このマークがついた atom の扱い |
| --- | --- | --- |
| `未検証` | 起草直後、本人未確認 | デフォルト。次の月次レビューで本人がチェックすべき対象 |
| `本人確認済` | 本人が事実を確認した | 安心して引用・参照してよい |
| `AI推測_要検証` | AI が推測で書いた、未確認 | **要注意**。引用前に本人確認を取る |
| `訂正済` | 過去に誤りがあり訂正した | `correction_history` に履歴あり |
| `削除推奨` | 内容に問題があり、`status: 廃版` への移行を検討中 | ビュー層から参照を外し、本人判断を待つ |

AI（抽出エージェント）が新規に作る atom は、原則として `verification_status: AI推測_要検証` で登録する。本人確認後に `本人確認済` に昇格させる。

### 0.4. 本文の書き方

- **冗長を恐れず、簡潔を最優先**しない。本文はその atom の正本なので、十分な情報量を持たせる
- 本人の語りスタイル（CLAUDE.md 準拠：「である」調優先、サンプル数1の留保あり）
- 引用は `> ` ブロックで明示し、出典を `*— XX.md より*` の形式で添える

## 1. Episode（EP）— 具体的に起きた一回の出来事

### 1.1. 役割

「いつ、どこで、何が起きたか」の記録。エピソード抽出の単位は**一回の出来事 + その出来事をめぐる解釈**。複数の似た出来事をまとめてはいけない（別 EP として登録する）。

### 1.2. フィールド

```yaml
id: EP-001
title: 海水パンツを引き上げてハイレグでへそを隠していた
status: 整理済
confidence: 確信
age_range: 8-12          # 必須・出来事が起きた年齢
period: TP-001           # 必須・人生の時期 ID
location: 学校のプール   # 任意・場所
related_people:          # 任意・関係する人物 ID の配列
  - PP-002
related_traits:          # 必須・関係する特性タグ
  - メイレズビアン
related_claims:          # 必須・このエピソードが支持する主張
  - CL-003
related_concepts:        # 任意・関連する概念
  - CO-001
related_episodes:        # 任意・直接関係する他エピソード
  - EP-005
情動: 違和感             # 必須・主たる情動タグ
created: 2026-05-06
updated: 2026-05-08
source:
  - 自身のエッセイ/01_性同一性障害メイレズビアン/01-1_メイレズビアンとは/01-1-3_おへそを見られることへの強い抵抗感.md
tags:
  - メイレズビアン
  - 〜15歳
  - 身体
  - 違和感
---
```

### 1.3. 本文の構造

```markdown
# EP-001 海水パンツを引き上げてハイレグでへそを隠していた

## 何が起きたか
（事実の記述。第三者が再構成できるレベルで）

## 当時どう感じていたか
（その時点での感覚。後付けの解釈は次節で）

## 後から見るとどう解釈できるか
（現在の自己理解からの再解釈）

## 関連
- 主張：[CL-003 性自認が女性である](../claims/CL-003.md)
- 概念：[CO-001 メイレズビアン](../concepts/CO-001.md)
- 似たエピソード：[EP-004 身体測定で素肌を触られて怒った](EP-004.md)
```

### 1.4. 良い EP / 悪い EP

| 良い EP | 悪い EP |
| --- | --- |
| 一回の出来事に焦点 | 複数の似た出来事を混ぜている |
| 当時の感覚と現在の解釈を分けている | 後付け解釈が混入している |
| 関連 atom への参照が複数ある | 孤立している |
| 出典が明示されている | 出典不明 |

### 1.5. 抽出のスコープ

- 5歳以前の記憶も対象（断片であっても EP として登録）
- 反復的な出来事（例：何度も殴った）は **「典型一回」を EP として登録 + 反復していたという事実を本文に明記**
- 他人から聞いた話で本人が直接記憶していない場合は `confidence: 推測` にして本文で明示

## 2. Claim（CL）— 著者が抱く命題

### 2.1. 役割

「私は〜だ」「〜である」「〜と思う」と本人が宣言する命題。エピソードや事実から帰納された自己理解の単位。

### 2.2. フィールド

```yaml
id: CL-001
title: 私には承認欲求がほとんどない
status: 整理済
confidence: 確信
type: about_self        # 必須・about_self / about_world / about_others
supporting_episodes:    # 必須・このCLを支持するエピソード
  - EP-034
  - EP-035
  - EP-036
  - EP-038
counter_evidence:       # 任意・反証となりうる事実
  - メモ_20260408_社会性と立場利用の欲求について.md  # 部分的反証あり
related_hypotheses:
  - HY-002
related_claims:
  - CL-002  # 派生命題
aliases:                # 任意・同じ趣旨の別言い回し
  - 4段階目の欲求が薄い
  - マズローの承認欲求が機能していない
created: 2026-05-06
updated: 2026-05-08
tags:
  - 承認欲求
  - 確信
---
```

### 2.3. 本文の構造

```markdown
# CL-001 私には承認欲求がほとんどない

## 主張
（1〜2文の明確な命題）

## 具体的な現れ方
- 出世への関心がない（[EP-034](../episodes/EP-034.md)）
- 「使われる側でいい」と決めた（[EP-035](../episodes/EP-035.md)）
- ...

## 検証の経緯
（マズローの10項目との照合など、どう確かめたか）

## 反論への応答
（「それは強がりでは？」等への応答）

## 関連
- 仮説：[HY-002 配偶動機-地位獲得本能の連動仮説](../hypotheses/HY-002.md)
- 派生：[CL-002 立場や肩書への関心が薄い](CL-002.md)
```

### 2.4. CL の粒度

- **粗すぎる**：「私は変わった人間だ」 → 内容が広すぎて検証不能
- **適切**：「私には承認欲求がほとんどない」 → 検証可能、エピソードと結びつく
- **細かすぎる**：「私は SNS のいいねが嫌いだ」 → これは RP（反応パターン）の方が適切

CL は**一文で言える、検証可能な、自己理解の単位**。

## 3. Hypothesis（HY）— 自家製の仮説

### 3.1. 役割

「もしかしたらこういう仕組みではないか」という**説明仮説**。既存の理論で説明できない現象を、本人が独自に枠組み化したもの。

### 3.2. フィールド

```yaml
id: HY-001
title: 配偶動機-地位獲得本能の連動仮説
status: 整理済
confidence: 仮説
one_liner: 配偶動機が成立しないと、地位獲得本能（マズロー4段階目）が発動しない可能性
predictions:           # 必須・この仮説が予測する観察
  - 性的指向が成立していない人は承認欲求が薄いはず
  - メイレズビアン同士で承認欲求の薄さが共通するはず
supporting_episodes:
  - EP-034
  - EP-035
supporting_claims:
  - CL-001
conflicting_evidence:  # 任意・仮説に反する観察
  - メモ_20260408_社会性と立場利用の欲求について.md
related_theories:      # 任意・関連する外部理論
  - TH-001  # マズロー
  - TH-005  # 進化心理学一般
related_hypotheses:
  - HY-002
created: 2026-04-30
updated: 2026-05-06
status_note: アセクシャル友人への観察で部分的に支持
tags:
  - 承認欲求
  - メイレズビアン
  - 仮説
---
```

### 3.3. 本文の構造

```markdown
# HY-001 配偶動機-地位獲得本能の連動仮説

## 仮説の主張
（1段落で）

## 仮説が説明しようとする現象
（何を説明するために立てた仮説か）

## 仮説の構造
（モデル図、メカニズムの説明）

## この仮説の予測
（仮説が真なら何が観察されるはずか）

## 支持証拠
- ...

## 反証証拠・未解決の問い
- ...

## 関連する外部理論
- [TH-001 マズローの欲求段階説](../theories/TH-001.md)

## 仮説の現状
（status_note と同じ。確信度の現状認識）
```

### 3.4. HY と CL の違い

- **CL**：自分自身についての命題（検証可能、自分で確かめられる）
- **HY**：自分以外（または自分と他者の比較）についての説明仮説（自分一人では確証できない）

## 4. Concept（CO）— 造語・独自定義の用語

### 4.1. 役割

本人が**特定の意味で使う言葉**の定義集。一般語と意味が異なる場合や、本人が独自に作った言葉が対象。

### 4.2. フィールド

```yaml
id: CO-001
title: メイレズビアン
status: 整理済
type: 独自造語          # 必須・独自造語 / 独自定義 / 既存語の本人用法
short_definition: 生物学的男性、性自認女性、性的指向女性の三条件を満たす人
coined_by: 本人
first_appearance: 自身のエッセイ/01_性同一性障害メイレズビアン/01-1_メイレズビアンとは/01-1-0_メイレズビアンとは.md
aliases:                # 任意
  - AMABレズビアン
  - male lesbian
related_concepts:
  - CO-002  # 性同一性障害
related_claims:
  - CL-003
created: 2026-05-06
updated: 2026-05-06
tags:
  - メイレズビアン
  - 性
---
```

### 4.3. 本文の構造

```markdown
# CO-001 メイレズビアン

## 短い定義
（1文）

## 詳細な定義
（必要十分条件、境界事例）

## なぜこの言葉が必要か
（既存の用語では言い表せない理由）

## 由来・初出
（造語の経緯、最初に書いた場所）

## 類似語との違い
- AMABレズビアン との違い
- トランスジェンダー（FtM/MtF）との違い

## 用例
（本人がこの語を使った代表的な文の引用）
```

### 4.4. CO のリスト（既知のもの）

本人の既存素材から確認できる Concept 候補：

- メイレズビアン / AMABレズビアン
- 4段階目薄人間

- サンプル数1
- HSP的感受性
- 合理性駆動モード
- ヒューマニスト / 功利主義者 / 理想主義者（自分用法）

## 5. Value（VL）— 価値観の単位

### 5.1. 役割

本人が大切にしている**価値**の単位。VL は単独では存在せず、必ず**価値の体系**の中に位置づく。

### 5.2. フィールド

```yaml
id: VL-001
title: 合理性
status: 整理済
priority: 高           # 必須・高 / 中 / 低
manifestations:        # 必須・行動への現れ
  - BP-002  # 行動パターン参照
  - BP-005
related_episodes:
  - EP-008  # 社会主義のほうが良いと思った
  - EP-014  # 偉い人=お金持ちへの違和感
related_claims:
  - CL-005  # 私は合理性で世界を判定する
related_values:
  - VL-002  # 整合性
  - VL-003  # 全体最適
opposed_values:        # 任意・本人にとって対立する価値
  - 立場 / 序列
origin: 子供の頃から（具体的な形成期は不明）
created: 2026-05-08
updated: 2026-05-08
tags:
  - 合理性
  - 価値観
---
```

### 5.3. 本文の構造

```markdown
# VL-001 合理性

## この価値観の中身
（何を「合理的」と感じるか、何を「非合理」と感じるか）

## 行動への現れ
（具体的にどう振る舞うか）

## 形成の経緯
（思い当たる原因。不明なら不明と書く）

## 対立する価値
（誰かが何を優先したとき、自分は摩擦を感じるか）

## 関連
- 派生する主張：[CL-005](../claims/CL-005.md)
- 関連エピソード：[EP-008](../episodes/EP-008.md)
```

### 5.4. VL のリスト（既知のもの）

合理性 / 整合性 / 全体最適 / 本質・道理 / 人間の尊厳 / 公正さ / 自己理解の継続 / 成長 / 客観的自己評価

## 6. Fact（FA）— 履歴書相当の客観事実

### 6.1. 役割

本人の客観的な属性データ。**履歴書として使える粒度**で整理する。
解釈や感情を含めない。

### 6.2. フィールド

```yaml
id: FA-001
title: 1977年生まれ
status: 整理済
category: 生年月日       # 必須・生年月日 / 出身 / 学歴 / 職歴 / 家族構成 / 健康 / 居住地 / 経済 / etc.
date: 1977
related_periods:
  - TP-000   # 出生〜（時期参照）
created: 2026-05-08
updated: 2026-05-08
tags:
  - 履歴書
---
```

### 6.3. 本文の構造

```markdown
# FA-001 1977年生まれ

## 内容
（事実の本体。短く）

## 補足（必要なら）
（解釈ではなく、事実の補足情報のみ）
```

### 6.4. FA のカテゴリ（網羅対象）

| カテゴリ | 例 |
| --- | --- |
| 生年月日 | 1977年生まれ |
| 出身 | （出身地） |
| 家族構成 | シングルマザーの一人っ子 |
| 学歴 | 専門学校卒 |
| 職歴 | 新聞奨学生（15-24歳）／新聞販売店店長 / 市場勤務（31歳）／運送業3社（34-36歳）／新聞配達中心の生活（37歳〜） |
| 副業・現業 | Uber Eats、執筆、AI活用 |
| 健康 | うつ病既往（20代後半〜30代）、DNA検査結果（情報処理が遅い傾向） |
| 居住地 | （現住地） |
| 経済状況 | （生計の現状） |
| 趣味・習慣 | アニメ・漫画・小説、オーディブル、AI活用 |
| 公開ハンドル | hoehoe / annachloe2025 |
| 関連 URL | GitHub annachloe2025、SelfAnalysis サイト URL |

> **取扱注意**：機微情報も**隠さず**書くと CLAUDE.md で本人が宣言している。ただし将来的に公開範囲を絞る選択肢を残すため、`category` で機微度を識別できるようにしておく。

## 7. Person（PP）— 関係者・登場人物

### 7.1. 役割

本人の人生に登場する人物。本人とその人物との関係、関わったエピソード、関係の変遷を記録する。

### 7.2. フィールド

```yaml
id: PP-001
title: 母
status: 整理済
relationship: 母      # 必須・母 / 父 / 元同僚 / 友人 / etc.
status_now: 不明       # 任意・現存 / 故人 / 連絡途絶 / 不明
period_active:         # 関係していた時期
  - TP-000  # 出生〜
notable_episodes:
  - EP-016  # 漫画情報が入りにくかった
  - EP-028  # 一人っ子だから我儘ね
significance: 高       # 高 / 中 / 低
created: 2026-05-08
updated: 2026-05-08
tags:
  - 家族
  - 〜15歳
---
```

### 7.3. 本文の構造

```markdown
# PP-001 母

## この人物について
（本人との関係、性格、現在の関係性）

## 本人への影響
- ...

## 関連エピソード
- [EP-016](../episodes/EP-016.md)

## 残された問い
- ...
```

### 7.4. PP の運用

- 本名は記録しない（プライバシー配慮）。役割や仮名で識別
- 同名の別人がいる場合は通し番号（友人A / 友人B）
- 重要度の低い登場人物は EP の本文中に書くだけでよく、独立 PP にしない

## 8. Period（TP）— 人生の時期区分

### 8.1. 役割

本人の人生をいくつかの時期に区切ったもの。**9年×2サイクル**仮説など、本人の自己理解と直結する。

### 8.2. フィールド

## 13. atom 同士の参照ルール

### 13.1. 必須参照

| atom | 必ず参照すべきもの |
| --- | --- |
| EP | 関連する CL を最低1つ |
| CL | 支持する EP を最低1つ |
| HY | 支持する EP または CL を最低1つ |
| CO | この語が登場する CL を最低1つ |
| VL | 関連する EP または CL を最低1つ |
| FA | 関連する TP を最低1つ |
| PP | 関連する EP を最低1つ |
| TP | 含まれる FA / EP の代表を数件 |
| TH | この理論を使う CL または HY を最低1つ |
| IN | この影響が現れる EP または CL を最低1つ |
| RP | 該当する EP を最低1つ |
| BP | 該当する EP を最低1つ |

### 13.2. 双方向参照の維持

- A が B を参照したら、B からも A への参照をフロントマターに追加する
- 参照の同期は整合エージェント（`05_オーケストレーション運用設計.md`）が担当
- 手動運用時は `validate_data.py`（運用ルール参照）で定期チェック

## 14. status / confidence / verification_status の運用

### 14.1. status（編集段階）

| 値 | 意味 |
| --- | --- |
| ドラフト | 抽出済み、本文未整備 |
| 整理済 | 本文整備、相互参照済 |
| 保留 | 内容に未決着の問題あり、判断保留中 |
| 廃版 | 取り下げ。物理削除はしないが参照しない |

### 14.2. confidence（内容の確信度）

| 値 | 意味 |
| --- | --- |
| 確信 | 本人が高い確度で正しいと判断 |
| 仮説 | 説明仮説の段階。反証可能 |
| 推測 | 根拠が弱い、要検証 |
| 保留 | 判断停止 |

### 14.3. verification_status（事実検証の段階）

| 値 | 意味 | 移行条件 |
| --- | --- | --- |
| 未検証 | 起草直後、未確認 | 月次レビューで本人がチェック → `本人確認済` または `AI推測_要検証` へ |
| 本人確認済 | 本人が事実を確認した | 訂正が必要になったら `訂正済` へ |
| AI推測_要検証 | AI が推測で書いた箇所。本人未確認 | 本人レビュー後 → `本人確認済` または `訂正済` へ |
| 訂正済 | 過去に誤りがあり、訂正した。`correction_history` に履歴あり | 引用可。履歴の確認は推奨 |
| 削除推奨 | 内容に問題があり、廃版検討中 | 本人判断後 → `本人確認済` に戻す または `status: 廃版` へ |

### 14.4. 3つのフィールドの組み合わせ早見表

| 状態 | status | confidence | verification_status | アクション |
| --- | --- | --- | --- | --- |
| AI が新規作成直後 | ドラフト | （AI推定） | AI推測_要検証 | 本人レビュー待ち |
| 本人が承認 | 整理済 | 確信 / 仮説 | 本人確認済 | 通常使用可 |
| 本人が起草、推敲中 | ドラフト | 確信 | 本人確認済 | 推敲後 整理済 へ |
| 仮説段階（自分でも未確証） | 整理済 | 仮説 | 本人確認済 | 引用時は仮説と明示 |
| 訂正処理した | 整理済 | 確信 | 訂正済 | 履歴を残して使用可 |
| 内容を取り下げ中 | 保留 | 保留 | 削除推奨 | ビューから参照を外す |
| 完全に取り下げ | 廃版 | -- | -- | 参照しない（履歴は残す） |

## 15. 抽出の優先順位（実装フェーズ向け）

データ層を構築する順序として、依存関係から次が自然である。

```
1. CO（概念定義）       … まず語彙を固める
2. TP（時期区分）       … 時間軸の骨格を固める
3. FA（履歴書事実）     … 動かしようのない客観事実を固める
4. EP（エピソード）     … 既存の EP-001 〜 EP-042 を昇格
5. CL（主張）           … EP から帰納
6. PP（人物）           … EP から抽出
7. RP / BP（反応・行動） … EP の典型化
8. VL（価値観）         … CL から抽象化
9. TH（外部理論）       … 既存の言及を整理
10. HY（自家製仮説）    … 既存6つを昇格 + 新規
11. IN（影響）          … 作品・人物の遍歴を整理
```

詳細は `05_オーケストレーション運用設計.md` の抽出フェーズで定義する。

## 16. 検証ルール

データ層に対して、`hooks/validate_data.py` が次を機械的にチェックする。

- すべての atom に必須フィールドが揃っているか
- 参照先の atom ID が実在するか
- 双方向参照の整合性（A が B を参照しているなら B からも A を参照しているか）
- ID の重複がないか
- ファイル名と中身の id が一致しているか

検証が通らない atom は `status: 保留` に自動で変更され、修正キューに入る。
