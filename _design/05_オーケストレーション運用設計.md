# 05_オーケストレーション運用設計 — エージェント役割と連携

> 本書は、データ層とビュー層を構築・運用するための**役割分担とワークフロー**を定義する。
> 目的は、本人 hoehoe の素材から atom を抽出し、ビューを構成するという作業を、**複数のエージェントに分担させて並列・反復可能にする**こと。

---

## 0. なぜオーケストレーションするのか

体系化を一人（一エージェント）で全部やろうとすると、次の問題が出る。

1. 文脈が広すぎてエージェントが**思考の飽和**を起こす
2. 抽出と整合と編集を**一つのエージェントが同時にやる**と、判断基準がブレる
3. **並列化できる作業を逐次でやる**ため、時間がかかる
4. 失敗箇所の**切り分け**が困難（どの判断が間違っていたかわからない）

役割を分けることで、各エージェントは自分の判断軸だけを保ち、人間（hoehoe）はオーケストレーターとしてエージェント間の整合だけを見ればよくなる。

## 2. エージェントごとの役割

### 2.1. オーケストレーター（人間 + 監督エージェント）

**主体**：本人 hoehoe（最終判断）+ Claude（補助）

**役割**：
- 各エージェントへのタスク発行
- 出力レビュー、却下・差し戻し
- 設計書（_design/）の更新承認
- 月次でフェーズ進捗を確認、TASKS.md を更新

**判断基準**：
- 設計書の方針と矛盾しないか
- 本人の自己認識と一致するか
- 公開範囲（特に機微情報）の方針

---

### 2.2. 設計エージェント

**主体**：Claude セッション

**役割**：
- `_design/` 配下の設計書の起草と改訂
- スキーマ変更の提案
- 新しい atom タイプ・ビュー軸の提案
- 既存設計の矛盾発見と修正

**入力**：本人の要望、現状診断、他エージェントからのフィードバック
**出力**：設計書の差分、提案書
**頻度**：必要時（フェーズ開始時、設計上の問題発覚時）

---

### 2.3. 抽出グループ（複数の専門エージェント）

抽出は **atom タイプごとに専門エージェントを立てる**ことで、判断軸を統一する。

#### 2.3.1. EP抽出エージェント

- **入力**：MyEssay / MyConsiderations / 既存 EP-001〜042 / 対話ログ
- **タスク**：
  - 一回の出来事を識別する
  - EP-XXX としてフロントマター付きで起こす
  - 既存 EP との重複を避ける
- **判断基準**：「一回の出来事 + その時の感覚」が atom 化できる粒度であるか

#### 2.3.2. CL抽出エージェント

- **入力**：MyEssay 全文、エッセイ分析レポート、CL を支持する EP 群
- **タスク**：
  - 「私は〜である」「〜と思う」型の命題を抽出
  - 既に他の言い回しで登録されていれば aliases に追加
  - confidence をつける
- **判断基準**：本人が確信を持って宣言している命題か、推測の域か

#### 2.3.3. HY抽出エージェント

- **入力**：既存 06_仮説と理論 配下、MyConsiderations の哲学カテゴリ
- **タスク**：
  - 自家製仮説を識別、HY-XXX として整備
  - predictions（予測）を必ず明文化
  - supporting / conflicting evidence を整理
- **判断基準**：他者と本人の比較を必要とする説明仮説であるか

#### 2.3.4. CO抽出エージェント

- **入力**：MyEssay / MyConsiderations / 対話ログ
- **タスク**：
  - 本人が独自の意味で使う語、造語を抽出
  - 短い定義と詳細な定義を書き起こす
  - aliases を整理（メイレズビアン / AMABレズビアン など）
- **判断基準**：一般語の用法と意味が異なるか、独自に作った語か

#### 2.3.5. VL抽出エージェント

- **入力**：CL 群、エッセイ分析レポートの「価値観の体系」節
- **タスク**：
  - VL を独立 atom として整備
  - priority と関係（対立・派生）を構造化
- **判断基準**：「これを大切にしている」と本人が認める単位か

#### 2.3.6. FA抽出エージェント

- **入力**：09_付録/01_プロフィール.md、09_付録/02_簡易年表.md、本人へのヒアリング
- **タスク**：
  - 履歴書相当の客観事実を atom 化
  - 不足カテゴリは本人にヒアリング
- **判断基準**：解釈を含まず、第三者が確認可能な事実か

#### 2.3.7. PP抽出エージェント

- **入力**：EP 群、MyEssay
- **タスク**：
  - 登場人物を識別、役割で命名
  - 重要度（significance）を判定
- **判断基準**：本人の人生に与えた影響が EP 単独で書ききれない人物か

#### 2.3.8. TP抽出エージェント

- **入力**：09_付録/02_簡易年表.md、HY-002（9年×2サイクル仮説）
- **タスク**：
  - 人生の時期区分を atom 化
  - 各時期の key_episodes / key_facts を紐付け
- **判断基準**：その時期に固有の主題があるか

#### 2.3.9. TH抽出エージェント

- **入力**：エッセイ全文、考察ノート、対話ログ
- **タスク**：
  - 本人が言及する外部理論を atom 化
  - how_used を明示（本人がどう使っているか）
- **判断基準**：本人の自己理解に実質的に貢献している理論か

#### 2.3.10. IN抽出エージェント

- **入力**：07_作品と趣味/、MyEssay 02_自分の特性、対話ログ
- **タスク**：
  - 影響を受けた作品・人物・体験を atom 化
- **判断基準**：本人の思想・嗜好・行動を変えた影響源か

#### 2.3.11. RP/BP抽出エージェント

- **入力**：EP 群（複数の EP に同型の反応・行動が見られるか）
- **タスク**：
  - 反復的な反応・行動パターンを抽出
  - 該当する EP を全て紐付け
- **判断基準**：複数の EP に共通するパターンか（一回限りなら EP のみで十分）

---

### 2.4. 整合エージェント

**役割**：
- 重複検出（同じ内容が違う ID で登録されていないか）
- 矛盾検出（時期によって認識がズレている、CL同士が衝突している）
- 双方向参照の整合（A → B があるなら B → A も書かれているか）
- 用語統制（aliases の整理）
- 順序の整理（時期、エピソードの並び）

**入力**：抽出グループの出力全体
**出力**：差分提案、修正パッチ
**頻度**：抽出フェーズの後、定期的（月次）

**重要**：整合エージェントは**新規の atom を生成しない**。既存の atom の整理だけを行う。

---

### 2.5. ビューエージェント

**役割**：
- ビュー層のページを書く
- データ層への参照リンクを正しく張る
- 視点別の解説（ナラティブ）を加える
- ビュー間の重複を意識する

**入力**：データ層の atom 群、設計書 04（ビューカタログ）
**出力**：`docs/views/` 配下のページ
**頻度**：データ層が一定揃ってから（Phase A 以降）

**重要**：ビューエージェントは**データ層を変更しない**。情報が不足している場合は抽出エージェントに差し戻す。

---

### 2.6. 検証エージェント

**役割（形式検証）**：
- 必須フィールドの欠損チェック
- 参照先 ID の実在チェック
- 双方向参照の整合性
- ID 重複の検出
- ファイル名と中身の id 一致
- カバレッジ：MyEssay の全文がどこかで参照されているか
- トーン整合：CLAUDE.md の文体規約に沿っているか

**役割（誤情報・捏造検出）**：
本人 hoehoe からの懸念に応えて、**AI が捏造した可能性のある内容を機械的に検出**する機能を持つ。次の兆候をフラグ立てする：

| パターン | 検出ロジック |
| --- | --- |
| 過度に整った文章 | 一文の平均長・接続詞の使用頻度・句読点パターンが、本人の既存エッセイと統計的に乖離 |
| 出典なき断定 | 本文中に断定表現（〜である／〜だ）があるのに `source` が空 |
| 不自然に細かい数値 | 「31.5歳のとき」「214冊」のような、本人があえて言わない粒度の数値 |
| 出典 source 空白 | `source: []` または欠損 |
| 名前の捏造 | PP（人物 atom）に本人未確認の固有名詞 |
| 仮説の即断定 | confidence: 仮説 なのに本文で「明らかに」「間違いなく」を使用 |
| 引用句の不存在 | 引用ブロック（`>`）の出典ファイルが実在しない |
| 三本柱・強み等の NG表現 | CLAUDE.md §8.1 の NG リスト該当語の出現 |

これらに引っかかった atom は、`verification_status: AI推測_要検証` に**自動降格**し、修正キュー（TASKS.md）に追加される。本人レビューで `本人確認済` または `訂正済` に昇格できる。

**入力**：データ層 + ビュー層の現状
**出力**：違反リスト（修正キュー）、`verification_status` 自動降格の通知
**頻度**：抽出フェーズ末・月次・公開前

**実装**：将来的には `hooks/validate_data.py` で自動化する。第一段階では形式検証のみ実装、捏造検出は段階的に組み込む。

## 3. ワークフロー（抽出フェーズの典型）

```
[1] 設計エージェントが抽出スコープを定義
    例：今回は EP のうち「メイレズビアン関連」だけを処理する

[2] EP抽出エージェントが起動
    対象素材を読み込み、EP-XXX を起こす

[3] CL抽出エージェントが並列で起動
    EP 群から帰納される CL を抽出

[4] CO抽出エージェントが起動
    EP / CL に登場する未定義の用語を CO として整備

[5] 整合エージェントが起動
    [2][3][4] の出力を統合、重複・矛盾を検出

[6] 検証エージェントが起動
    必須フィールド・参照整合をチェック

[7] オーケストレーター（人間）がレビュー
    内容の真偽、機微情報の取り扱いを最終判定

[8] 承認された atom が docs/data/ に commit
```

---

## 4. ワークフロー（ビューフェーズの典型）

```
[1] 設計エージェントが今回作るビューを定義
    例：今回は「メイレズビアンレンズ」を作る

[2] ビューエージェントが該当 atom を集める
    タグ「メイレズビアン」が付いた全 atom を抽出

[3] ビューエージェントが atom を物語として並べ、
    視点別の解説を加える

[4] 整合エージェントがチェック
    既存ビューとの重複、データ層への参照が正しいか

[5] 検証エージェントがチェック
    リンク切れがないか

[6] オーケストレーター（人間）が読み返してレビュー

[7] 承認後 docs/views/ に commit
```

## 5. エージェントへのプロンプトテンプレート

各エージェントへ依頼するときに、最低限揃えるべき情報を以下に示す。実際の起動時はオーケストレーターが Claude のサブエージェントツールに渡す。

### 5.1. EP抽出エージェントへの依頼テンプレ

```
あなたはエピソード抽出専門のエージェントです。

【目的】本人 hoehoe の素材から、atomic な「一回の出来事」を抽出して
EP-XXX として整備する。

【入力素材】
- C:\Users\hoeho\Documents\Claude\MyProfile\MyEssay\<対象フォルダ>
- 既存 EP: docs/data/episodes/ にある EP-001 〜 EP-042
- 設計書: SelfAnalysis/_design/03_データ層スキーマ仕様.md の §1

【スコープ】
今回は「<対象テーマ>」に関連するエピソードのみを対象とする。

【判断基準】
- 「一回の出来事」を一つの EP に。複数の似た出来事は別 EP として登録
- 既存 EP-XXX で扱われている内容は登録しない（重複検出）
- 当時の感覚と現在の解釈は明確に分けて書く
- confidence は本人がどれだけ確信しているかで判断

【出力】
- 新規 EP-XXX.md を docs/data/episodes/ に作成
- 各 EP に必須フィールドを揃える
- 関連 CL / CO / RP / BP / PP がデータ層にあれば参照を張る
- 関連 atom がデータ層にない場合は本文中に「[未登録: 概念名]」とマーク

【完了条件】
- スコープ内の素材を読了
- すべての候補 EP を起こした（または既存と重複と判断）
- 整合エージェントへの引き継ぎメモを書いた
```

### 5.2. ビューエージェントへの依頼テンプレ

```
あなたはビュー作成専門のエージェントです。

【目的】データ層の atom を <視点> で束ね直し、読み物として構成する。

【入力】
- データ層: docs/data/ 全体
- 設計書: SelfAnalysis/_design/04_ビュー層カタログ.md の §<該当節>
- 関連タグ: <タグ名>

【作業】
1. タグ <タグ名> が付いた atom を全て収集
2. 視点別の解説（ナラティブ）を冒頭に書く
3. atom を構造化してリンク列挙する
4. ビュー末尾に「関連ビュー」を3つ程度貼る

【禁止事項】
- 本文を書く（atom の内容を引用するのは可、新規本文は不可）
- atom を新規作成する（不足は整合エージェントに差し戻す）

【完了条件】
- ビューファイル docs/views/<軸>/<ビュー名>.md を作成
- 検証エージェントへの引き継ぎメモを書いた
```

## 6. エージェント間の引き継ぎ規約

各エージェントは作業終了時に**引き継ぎメモ**を残す。`_design/handoff/` に置く。

引き継ぎメモのテンプレート：

```markdown
---
from: EP抽出エージェント
to: 整合エージェント
date: 2026-05-08
session_id: 2026-05-08-ep-batch-A
---

# 引き継ぎメモ

## 今回の作業
- 対象スコープ: メイレズビアン関連 EP の抽出
- 処理した素材: MyEssay/01_性同一性障害メイレズビアン/01-1_*

## 成果
- 新規作成: EP-043, EP-044, EP-045
- 既存更新: EP-002（出典追加）
- 重複と判定して登録しなかった: 3件

## 残課題・要判断
- 「ハイレグでへそを隠した」の体験は EP-001 で既出だが、
  「中学時代も同じ行動を続けた」という記述は別 EP にすべきか判断保留
  → 整合エージェントの判定をリクエスト

## 提案
- 「身体への過敏さ」は反復的なので、新規 RP（反応パターン）を立てるべき
  → RP抽出エージェントに依頼推奨
```

## 7. オーケストレーター（人間 hoehoe）の判断ポイント

人間が**必ず判断する**必要がある場面を明示する。AI に任せきりにしてはいけない場面である。

| 場面 | なぜ人間判断が必要か |
| --- | --- |
| 機微情報の公開可否 | プライバシー、家族・関係者への配慮は本人にしか判断できない |
| 自己認識の確信度 | 「これは本当に自分のことか」の最終判断は本人 |
| 仮説の採否 | エージェントが立てた仮説に違和感があれば棄却 |
| 用語の好み | 「メイレズビアン」と「AMABレズビアン」のどちらを正本にするか等 |
| ビューの統廃合 | どのビューを残すか、どのビューを統合するかは戦略判断 |
| 公開範囲の調整 | データ層の機微情報をビュー層でどこまで露出させるか |

## 8. オーケストレーション運用の段階

### 段階1：手動運用（現フェーズ）
- 人間がオーケストレーターを兼ねる
- Claude セッション一つを「設計エージェント」として動かす
- 抽出は Claude のサブエージェント（Agent ツール）で並列起動

### 段階2：半自動運用
- `hooks/validate_data.py` で検証を自動化
- 引き継ぎメモを構造化し、エージェント間で機械的に渡せるようにする
- 監督エージェントを Claude のサブエージェントとして常駐

### 段階3：自動運用（将来）
- 新しい考察ノートが生まれたら、抽出エージェントが自動で atom 化を提案
- 月次でビューを再生成
- 人間は採否判定のみに関与

---

## 9. オーケストレーションの失敗パターンと対策

| 失敗 | 対策 |
| --- | --- |
| 抽出エージェントが atom を作りすぎる | スコープを明示、上限を設ける |
| 抽出が雑になる | 設計書03のスキーマを必ず参照させる |
| 整合エージェントが新規 atom を作る | 「整合は新規生成しない」を厳守 |
| ビューエージェントが本文を書いてしまう | 「ビューは参照のみ」を厳守 |
| 検証が形式だけで内容を見ない | 内容検証は人間が担う、機械検証はあくまで形式 |
| 人間がレビュー疲れする | フェーズを区切る、一度に一軸だけを処理する |

---

## 10. このフェーズでの Claude の使い方の指針

- 設計（このフォルダ）は **Opus / Sonnet クラス** が適切
- 抽出は **Sonnet** で十分（タスクが定型化されている）
- 検証は **Haiku** でも可（機械的なチェックが中心）
- 整合は **Sonnet** が必要（判断を伴う）
- ビュー作成は **Sonnet / Opus**（語りの質を求める）
