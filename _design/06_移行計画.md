# 06_移行計画 — 現状から新アーキテクチャへ

> 本書は、現在の SelfAnalysis フォルダから 3層アーキテクチャへの**段階的移行手順**を定義する。
> 既存資産を捨てずに、段階を踏んで再構築する。

---

## 0. 移行原則

1. **既存ファイルは削除しない**。`_archive/` か `docs/_legacy/` に退避させ、可逆性を担保する
2. **段階ごとに動く状態で commit する**。途中でビルドが壊れることを許さない
3. **ビューを先に作らない**。データ層が一定揃うまではビューに着手しない
4. **公開サイトは並行運用**する。新構造ができるまで旧構造を表示し続ける
5. **本人レビューを必ず挟む**。各フェーズの完了は本人 hoehoe の承認をもって判定する

---

## 1. 移行のフェーズ全体像

```
Phase 0：準備（設計書承認、退避先作成）
   ↓
Phase 1：データ層スケルトン（フォルダと最初の atom 数件）
   ↓
Phase 2：既存 EP 昇格（EP-001〜042 をデータ層に移動）
   ↓
Phase 3：その他の atom 抽出（CL/CO/HY/VL/FA/PP/TP/TH/IN/RP/BP）
   ↓
Phase 4：整合と検証（重複検出、参照整合、機械検証導入）
   ↓
Phase 5：ビュー層構築（最小10ビュー）
   ↓
Phase 6：旧構造の整理（_legacy/ への退避、孤児ファイル処理）
   ↓
Phase 7：拡充ビュー（追加10ビュー）
   ↓
Phase 8：運用ルール本格化（hooks 整備、自動検証、月次運用）
```

各フェーズは数日〜数週間。並列可能な作業は並列化する。

## 5. Phase 3：その他の atom 抽出

### サブフェーズと優先順位

依存関係を踏まえ、以下の順序で進める。

#### Phase 3.3：関係的 atom（並列可能）
- **PP 抽出**（人物）

  - 推定件数：5〜15件
- **TH 抽出**（外部理論）

- **IN 抽出**（影響源）

## 14. 並行運用戦略（公開サイトを止めない）

移行中も公開サイト（GitHub Pages）を止めないために、以下を徹底する。

- 旧構造は `_legacy/` ではなく `docs/legacy/` に置く（mkdocs ナビには載せない）
- `mkdocs.yml` の nav は新構造のみを表示
- 旧 URL からのリダイレクトは設定しない（公開直後なので参照者は少ないと想定）
- 大規模な構造変更は休日深夜に実施し、ビルド失敗時は revert で即復旧

---

## 15. ロールバック方針

各フェーズで失敗を検出した場合の戻し方を明示する。

| 失敗の種類 | 戻し方 |
| --- | --- |
| ビルドエラー | 直前のコミットに `git revert` |
| データ層の構造ミス | 該当フェーズの commit を revert、設計書を修正してから再開 |
| 本人レビューで NG | 設計書を改訂、影響範囲の atom / ビューを再生成 |
| 機微情報の誤公開 | 即座に `_archive/` へ移動、`git filter-repo` で履歴から除去 |

- [ ] データ層の 12タイプすべてに atom が存在し、相互参照が整っている

- [ ] 旧 `docs/` 配下の二重化が解消されている
- [ ] 機械検証が CI で通っている

- [ ] 本人 hoehoe が「これで完全に分解できた感じがする」と判断
